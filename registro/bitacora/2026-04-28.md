### Decisiones técnicas:

#### 1. Realtime — `@supabase/ssr` `createBrowserClient`
- **Problema**: El browser Supabase client se creaba con `createClient` crudo (sin auth). RLS en `notifications` (`auth.uid() = user_id`) bloqueaba todos los eventos Realtime porque el client no tenía sesión.
- **Intentos fallidos**:
  - `setSession()` manual con tokens del loader → race condition (async vs sync hook mount)
  - Promise gate para esperar auth → no resolvía el problema de fondo
- **Solución final**: Usar `createBrowserClient` de `@supabase/ssr` que lee auth de cookies automáticamente (misma estrategia que el server client). Zero config, zero race conditions.

#### 2. Web Push — Encriptación ECDH + AES-128-GCM
- **Problema**: El payload se enviaba sin encriptar. Web Push (RFC 8291) requiere encriptación obligatoria. FCM rechazaba silenciosamente los mensajes.
- **Solución**: Migración a `@negrel/webpush` (JSR, compatible con Deno/Edge Functions). Implementa VAPID + encriptación RFC 8291 nativamente.
- **Columnas DB**: Corregido `p256dh`/`auth` → `keys_p256dh`/`keys_auth` (nombre real en `push_subscriptions`).

#### 3. VAPID Keys — Regeneración y alineamiento
- **Problema**: VAPID keys existían solo en Cloudflare Workers secrets, no accesibles desde Supabase Edge Functions.
- **Solución**: Nuevo par VAPID generado. Almacenadas en:
  - Cloudflare Workers: `wrangler secret put`
  - Supabase: tabla `app_config` (RLS sin policies públicas, solo service_role)
  - Frontend: hardcoded en `use-push-notifications.ts`
- **Migración**: `app_config` table creada, suscripciones viejas purgadas (incompatibles con nueva VAPID key).

#### 4. Página /notificaciones — Contexto compartido
- **Problema**: La página creaba su propio `useNotifications()`, generando una suscripción Realtime duplicada que no se sincronizaba con el Orb del layout.
- **Solución**: El layout pasa `notifs` via `Outlet context`. La página consume directamente `ctx.notifs` — misma instancia, misma data, cero duplicación.
